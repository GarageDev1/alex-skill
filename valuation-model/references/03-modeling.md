# Step 3 — 建模通链 ★(本 skill 的核心,旧版翻车正中靶心)

> 用公式连接物理锚、营收、利润、估值和隐含价格。这是模型能否正常工作的主要验收项。
> **★ 例 ≠ 规范**:同 `04` 开头约定——本文件带具体数字/标的名的例子(标"如/例",多来自海力士 worked example)只是说明原则,新标的一律按原则现拉现算,禁照抄例子数字。

## 链骨架(公司无关)
```
[① 物理锚] → [② TAM/需求] → [③ 营收] → [④ 毛利→净利→账面/FCF] → (交给 Step 4 估值)
```
- ①②③ 的展开因物理锚而异,见 `anchors/<类型>.md` 的 build pattern。
- ④ 是旧版**完全缺失**的一段:营收必须接着推到毛利、净利、账面价值 —— 不能停在营收就甩给估值。

## 三条公式连通要求
**A. 每一节输入 = 上一节输出(公式引用)。**
不许某一节直接硬塞一个卖方数字把链截断。典型反面:营收模型建完,估值却用 MS 的净利润 —— 链断。

**B. 无孤儿(No Orphan)。**
每个算出来的 driver 行/sheet,必须有 ≥1 个下游公式引用它。算了没人用 = 删掉或接上。
典型反面:AIDC 拆解表算出 HBM=$10.5B/GW,却没有任何公式引用它 → 应接成 TAM 的来源。

**C. 基年真实。**
③④ 的投影起点 = Step 2 历史财务的真实值,不许 plug/反算。
典型反面:`传统DRAM 2025 = 总营收 − HBM − NAND − 其他`(倒挤)。

## ③ 营收的三种通用写法(段级选用, 同一公司可混用)
- **份额式**(平台/消费/SaaS):`营收 = TAM(来自②) × 公司份额 × 单价`。
- **强度式**(物理锚直驱段——需求由基建/资本开支拉动的部分,如 AI 算力链):`段收入 = 物理锚 × 收入强度`。强度 = 该段收入 ÷ 物理锚,**锚最近实际年公式反推**(只锚 1 个真实数),前瞻从锚平滑续上;历史列公式反推、锚未成规模的早年标 n.m.。三条铁律见 `anchors/<类型>.md`。
- **周期式**(无直驱的消费/商品段):`段收入 = 上年 × (1+量增速) × (1+价格变化)`,价格走行业供需缺口的周期判断,缺货年量可为负(产能被直驱段挤占 + 涨价需求破坏)。
- **★ 供需取 min 不再作单公司收入的主驱动(已废)**:旧版"供给=产能×单片产出×ASP×份额 与 需求 取 MIN"被砍,因为 ① 产能本质由建厂周期驱动,"产能=锚×强度"是强行连通的 hack(概念错);② 四个粗数连乘,误差相乘失真。供需缺口只用于**行业价格建模**(给周期式的价格路径供弹药);公司产能够不够,用管理层指引做**旁注可行性检查**,不挂主链。

## ★ 每个前瞻假设行必须配「历史实际列」(锚 + 回测)
任何前瞻假设(bit 增速 / 价格变化 / 利润率 / 留存率 / 分部占比 / 收入强度…),在 Excel 里**必须把 2021-2025 历史实际值排在同一行的左侧**,前瞻值接在右边——不能只有前瞻列。
- **为什么**:① 给读者一个"锚"——前瞻 50% 毛利率合不合理,要看历史走过 44/35/−2/48/60% 才知道;② 让前瞻能被历史**回测**(前瞻第一年与最近实际年要平滑衔接,不跳断层,见 `anchors` 强度反推铁律);③ 暴露周期振幅——利润率历史走过 +29%→−24%→+49% 这种剧烈周期,任何"稳态假设"一看历史就破。
- **做法**:假设 sheet 的利润率/增速/占比行用 ALLC(2021-2030)而非只 FC(前瞻);历史列填**实际值**(标蓝或灰、注明"实际"),前瞻列填假设(蓝字)。分部级历史不披露的(如 HBM 段毛利率)填卖方估 + 标"估",早年极小的标 n.m.。
- **判断标准**:任何前瞻假设的左侧都要展示该指标的历史区间。过往利润率和留存率只给出前瞻列,缺少历史参照,导致假设无法评估。

## ④ 营收 → 盈利(★ 故意保持粗颗粒,见 SKILL"模型海拔")
这一段只干一件事:把物理驱动出来的**营收**转成估值要用的盈利口径。**不做三表、不做营运资本、不逐科目拆 opex**——会计精度不是卖点。颗粒度两档,按业务选:
```
【默认: 一根线】分部盈利结构同质时
净利   = 营收 × 净利率假设            # 净利率接历史区间/可比, 分情景给, 当假设处理

【升级: 段驱动】分部盈利结构性分化时(如 定制/franchise 段 vs 商品段, 利润率差几十个点)
营业利润 = Σ(段收入 × 段营业利润率)    # 段OPM 各自锚历史实际/最新季报, 分情景给
净利     = 营业利润 × 净利转换率        # 税+利息等一根转换线, 锚历史实际

─ 给 P/E:  EPS = 净利 / 当年股本
─ 给 DCF:  FCF ≈ 净利 × FCF转换率     # 同样是假设, 不拆 D&A/CapEx/NWC
─ 给 P/B:  期末账面 = 期初账面 + 净利 × 留存率   # 只在用 P/B 时才算, 仅此一行留存递推
```
**纪律**:
- **净利率/段OPM/净利转换率/FCF转换率 都是假设**,落到假设 sheet、配历史实际列、分 Bear/Base/Bull、带 source。不要假装它们是精算出来的。
- **段驱动是三情景挂"段利润率杠杆"的前提**:情景故事里若"定制段定价权稀释"和"商品段周期回落"是两条不同的路径(翻档幅度不同),就必须升级到段OPM——一根线挂不住两个故事。
- **链要连通**:净利必须由公式从上游营收/段收入算出,**不许直接抄卖方净利序列**(那会把物理驱动的链截断 —— 旧 Hynix 正是抄了 MS 的净利,业务模型成摆设)。
- 拆得更细(毛利-opex 两段、逐科目)只在**它影响方向性判断**时才拆;否则到段OPM 为止。

## 连通性自检(写完模型立刻跑 `scripts/validate_valuation.py`)
脚本机械检核心项,任一 **fail** → 回本步重写,不能进 Step 4:
1. **连通性(到物理锚)**:隐含价输出格的依赖链能否回溯到基座 sheet。不能 = 链断。
2. **孤儿扫描**:中间 driver 是否被下游公式引用。算了没人用 = 接上或删。
3. **溯源覆盖**:蓝字硬编码输入行是否有可见逻辑/备注列。
4. **禁单元格批注**:全簿不得用红三角批注藏来源。
5. **多镜头 + 三情景结构** 是否齐。

补做一项脚本无法覆盖的人工检查:**修改 ① 物理锚的一个输入 → recalc → 检查隐含价是否变化**。

### 连通性同时检查公式可达和数值灵敏度
`validate_valuation.py` 的可达性检查是必要但不充分的。它只检查隐含价的公式依赖链能否回溯到锚 sheet,无法判断修改锚的数值后价格是否变化。Hynix 模型中,`实际出货 = MIN(供给, 需求)`,需求侧引用 capex,供给侧产能则为定值。在供给受限的年份,MIN 始终输出供给,因此修改 capex 不会改变 HBM 收入和估值,但可达性检查仍会 PASS。
- **`MIN`/`MAX`/`IF`/clamp 会数值上切断物理锚**:当锚只驱动被夹掉的那一侧时,锚就成了摆设。
- **补充验收要求**:模型完成后必须人工检查数值灵敏度。将某年物理锚调整 ±20% 并重算,确认隐含价随之变化;否则即使可达性 PASS,模型仍不合格。
- **设计上**:单公司收入主链尽量避开 `MIN/MAX/clamp`。供需缺口放在行业价格路径或可行性旁注里;若某类业务确实必须用约束函数,必须单独做数值灵敏度测试,证明约束没有把物理锚夹成摆设。

> 旧 Hynix 模型的实测结果为:validate 连通性 0/11 fail(AIDC 表未被引用),溯源 3% fail。本流程需要解决这两项问题。

## ★★ 逻辑闭环验证(建完模型后必做的最后一步)
拍完所有参数、数字都跑通后,**不能就交付**——还要回头检查**模型派生出来的比值/份额,是否和你的 thesis 同向**。一个拍得准的模型,怎么看都应该是逻辑自洽的;如果某个派生比值的走势和 thesis 相反,说明内部有矛盾(某段拍太松/太紧),回去改。
- **做法**:挑几个 **thesis 直接隐含方向**的派生比值(分部 mix 份额、AI 段占比、利润率结构…),拉成一行看逐年走势,对照 thesis 应有的方向。把它做成模型里一行**可见的「闭环检查」**,让读者也看到自洽。
- **实测问题**:thesis 认为 AI 服务器将吸收 DRAM 增量,server 占传统 DRAM 的份额应逐年上升。原模型中,server 使用 `capex×强度`(受 capex 增速限制),consumer 使用 `bit×价`(完整反映涨价),结果 server 份额从 50% 降至 41%,与 thesis 相反。这说明 intensity 法低估了涨价年的 server 收入。
- **修法范式**(同一现象的两个口径要对齐):① commodity 段的 `收入/capex` 强度**不是常数**,它含了内存价 → 涨价年强度冲高、价 normalize 后回落(只有 HBM 因价更稳、份额驱动,强度才近似稳定);② 缺货年 consumer 的 bit 要**压成负**(涨价的需求破坏 + 厂商把产能转去服务器/HBM)。修对后 server 份额回到上升,闭环成立。
- **注意口径**:闭环检查要挑**对的比值**。如"server vs consumer 占传统DRAM"(同价、比 bit 分配)→ AI 吃增量 → 必升,是有效检查;而"HBM 占总DRAM 收入%"会被商品段的价格超级周期暂时摊薄(2026 商品 ASP +176% 反而盖过 HBM),所以它涨价年下降是正常的、不算违反 thesis。**别拿被价格扭曲的收入份额当 bit 逻辑的闭环检查。**

### ★ 分部占营收 % 要逐年列出来看,但分清两类份额(bit vs 收入)
**预测表必须有一行"分部收入占营收 %"**(逐年),让读者看到收入结构怎么演化。但读它时要分清两类份额、用途不同:
- **bit/量 份额**(如 server 占传统DRAM bit):反映 thesis 方向(AI 吃增量)→ **必须同向**,是有效闭环检查。
- **收入 份额**(如 HBM 占总营收):会被**价格周期扭曲**——涨价年商品段(传统DRAM/NAND)收入被 +130~180% ASP 灌爆,即便 HBM bit 份额在升,HBM 收入% 也会**暂时下降**。这**不是 bug**(GS/Citi 同样显示 HBM 收入% 在涨价年下降),不能当 thesis 违反。
- **实测**:用户看到"HBM 收入占营收 2025→2026 从 29% 降到 19%"以为是 bug——其实是 2026 商品价超级周期(传统DRAM 收入 +159%、NAND +173% vs HBM +53%)把 HBM 收入% 摊薄,与 GS(conventional DRAM +247%)一致。
- **做法**:列"分部收入占营收%"行 + 配一句注明"收入占比受价格扭曲,thesis 方向看上方 bit-份额闭环检查";并在大异动检查里**逐段验证 YoY 增速对得上卖方**(传统DRAM +159% < GS +247% 即偏保守、合理)。
- 判断标准:每个有明确 thesis 方向的派生比值,逐年走势都和 thesis 同向 → 闭环;反向 → 模型有内在矛盾,必修后才交付。

### ★ 闭环检查还要盯「关键指标的大异动」(不只看方向,还看幅度)
方向对了还不够。**任何关键 P&L 行(营收/营业利润/净利/EPS)单年 YoY 摆动 >±30%,都要过三关**:
1. **有意的驱动**:能追到一个明确原因吗(如周期价格见顶、产能投产)?追不到 = 可能是 bug。
2. **幅度对得上历史**:对照这家公司历史上类似拐点摆了多少。差太远 = 参数拍过头。
3. **不是参数悬崖/公式人工痕迹**:是不是某个假设拍了个单年"悬崖值"(如价格单年 -30%)人为造出来的跳变?
- **实测**:营业利润 2027→2028 从 115 跌到 68(-40%)——追得到驱动(2028 内存价 normalize → 营收 -18% + 商品段经营杠杆把 margin 58%→34%),幅度也对得上历史(比 Hynix 真实 2023 转亏还温和),**但第 3 关没过**:是单年价格 -30% 的悬崖拍出来的。真实下行周期是 2-3 年缓慢回落,不是一年跳水 → **把 normalize 摊到几年滑落**,OP 曲线就平顺了。
- 判断标准:大异动要么追到有意驱动 + 幅度合历史 + 不是人为悬崖,**三关全过才放行**;否则平滑或修。别让未经检验的大跳变直接进交付。
