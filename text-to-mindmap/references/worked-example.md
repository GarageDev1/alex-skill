# 参考实录:一张原图到这套设计怎么来的

这份 skill 的设计不是拍脑袋定的,是从一次真实任务里三轮返工收敛出来的。记下来,是为了让后来者别再踩一遍同样的坑。

## 起点
用户丢来一张竖版长图:光互联产业的树形大纲(光模块/NPO/CPO、产业逻辑、厂商布局、国产替代),含两张对比表和一张象限散点图。要求:摘出内容,做一张**更精美的思维导图**,输出图片,保持分辨率。

## 第一版(被否):七彩卡片
做法:7 个分类各一种颜色,每条要点一张带**彩色左边框**的卡片,术语用**彩色文字**,表格里高/中/低用**红/橙/绿标签**。
用户反馈:**"风格不行,不够清晰,颜色排版混乱。"**
病根:一屏里同时出现七八种颜色 + 红橙绿标签 + 彩色术语,全在抢注意力,没有主次。颜色被当成了装饰,而不是信息结构。

## 第二版(方向对但偏题):单色系收敛
做法:砍到**单一靛蓝主色 + 中性灰**,层级靠字号/留白;表格改**同色系等级条**;编号做成 01–07 徽标 + 竖直主脊。
结果:干净了,但用户要的是**原图那种逻辑树连接**——"一个线,一个线,一个点,然后一块连起来",不是我自作主张的编号列。偏题。
教训:清晰度问题解决了,但**别改用户已经认可的结构**。用户没否定原图的布局,只否定了我的配色。

## 第三版(定稿):逻辑树 + 单色系纪律
做法:回到原图的**竖向逻辑树**——主干竖线 → 彩色分类节点(圆环+横线) → 子干 → 每条要点用横线+圆点连到内容块。同时保留第二版的配色纪律:**一个分类一种颜色,分类内不混色**;表格等级条;重点行浅色底。
结果:通过。这就是现在 `assets/template.html` 的 `business-blue` 主题。

## 沉淀成 skill 时的关键决定
- **输入是文本,不是图**:skill 不做"重制原图",而是"把内容做成思维导图"。给散文先归纳成结构化 JSON 再出图。
- **布局唯一,主题多套**:用户要的"多种风格"是**美术风格**(配色/字体质感),不是多种树形拓扑。所以一套逻辑树 + 3 套主题皮。
- **中间层固定 JSON schema**:表格、象限图都是一等公民节点类型,渲染确定、可复现,不会每次手搓漂移。
- **事实忠于输入**:思维导图是把用户给的内容重新组织,不发明新事实、不外部取数;缺了就回问。
- **强制自检**:HTML 截图最容易出"看着精美但文字被切一块"的无声故障,交付前必须 Read 回看逐区核对。

## 一句话
颜色是用来分组和指示重点的,不是用来装饰的;结构照用户认可的来,别自作主张换骨架;能确定性复现的东西就别靠每次手搓。
