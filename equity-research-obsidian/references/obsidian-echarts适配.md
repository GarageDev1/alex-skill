## 一、全 inline ECharts 完整骨架(每张图必须自带 layout)

每个 `echarts` 代码块都必须是**完整、合法、可独立渲染的 JSON**。不得依赖格式化脚本、CSS 注入、其他 Skill 或任何外部自动处理来补 layout。每张图至少显式写出 `width`、`height`、`title.subtext`、`tooltip`、`legend`、`grid`、坐标轴和 `series`;无图例的单序列图也保留 `legend` 并可设空数据。

````markdown
**图表 N：<中文描述>**

```echarts
{
  "width": 900,
  "height": 506,
  "backgroundColor": "transparent",
  "textStyle": {
    "fontFamily": "Arial, Microsoft YaHei, sans-serif",
    "color": "#333"
  },
  "title": {
    "subtext": "数据来源：XXX / YYYY-MM-DD | 关键数字",
    "left": "center",
    "top": 8,
    "subtextStyle": { "fontSize": 12, "color": "#666" }
  },
  "tooltip": {
    "trigger": "axis",
    "confine": true,
    "backgroundColor": "rgba(255,255,255,0.96)",
    "borderColor": "#ccc",
    "textStyle": { "color": "#333" }
  },
  "legend": {
    "top": 48,
    "left": "center",
    "type": "scroll"
  },
  "grid": {
    "left": 70,
    "right": 35,
    "top": 95,
    "bottom": 65,
    "containLabel": true
  },
  "xAxis": {
    "type": "category",
    "data": ["2024A", "2025A", "2026E"],
    "axisLabel": { "color": "#555" },
    "axisLine": { "lineStyle": { "color": "#aaa" } }
  },
  "yAxis": {
    "type": "value",
    "name": "单位",
    "axisLabel": { "color": "#555" },
    "splitLine": { "lineStyle": { "color": "#eee" } }
  },
  "series": [
    {
      "name": "主公司",
      "type": "line",
      "data": [1, 2, 3],
      "smooth": false,
      "symbolSize": 7,
      "lineStyle": { "width": 3, "color": "#d33" },
      "itemStyle": { "color": "#d33" }
    }
  ]
}
```

> 📂 **资料来源**：{具体来源 URL / 公司 IR / 第三方数据库}
````

图下**只留一行来源**(像券商研报"资料来源：公司公告，XX 研究所")。**不写「📊 看图要点」**——对这张图说明了什么的解读,由**图前的一段正文**承担(见 §三的图文混排),不在图底单挂一句标签。

## 三、Chart 命名 + 图文混排(图作佐证,解读入正文)

ICA 阶段 5 强制"图表 N 全文连续编号、标题在上、来源在下"。Obsidian 版落地:**每张图一个 `**图表 N：<中文描述>**` 加粗图注 + ECharts 代码块 + 一行资料来源**。

> **图表标题用加粗文本,不用 `###`(或任何 `#`/`##`/`###`)标题层级。** 图表标题是图的 caption,不是文档结构——用 `###` 会让"图表 1""图表 2"……几十个图注涌进 Obsidian 大纲(outline),和真正的章节、叙述小节标题同级混杂,把文档结构淹没。加粗图注视觉上仍醒目、保留连续编号,但不进大纲。

### 3.1 命名与图注

- **编号连续不断号**(图表 1、图表 2……图表 70),不分图和表两套号。40-70 张全文连续。
- **加粗图注 = 图表 N + 中文描述**,带判断/画面(如 `**图表 11：腾讯分业务营业收入（亿元）**`),不写 ⭐/"Figure N"。
- **ECharts 内不写 `title.text`**(§二)。
- **代码块后只跟一行来源 blockquote**:`> 📂 **资料来源**：<具体来源>`。**不写「📊 看图要点」**——对图的解读由图前正文段落承担,不在图底单挂一句标签。
- **图表标题用加粗文本、不占任何标题层级**,因此不进 Obsidian 大纲;只有真正的章节(`##`)、叙述小节(`###`)才用标题层级。叙述里要配图就由前一段话引出、图紧跟该段(仍带加粗图表 N 图注)。

### 3.2 图文混排范式(正常研报的样子)

研报的图是**论证的佐证**,不是论证之后的附录。目标形态是**段→图→段→图交替前进**:每张图前先有一段描述性散文,把这张图要说明的观点和关键数字讲清楚(这段文字本身就是对图的解读),然后紧跟引出的那张图佐证。读者顺着正文读下来,图在该出现的地方出现。

- ✅ **推荐**:`一段论证文字(讲透观点 + 关键数字,自然引出"见图表 N")` → `紧跟图表 N` → `下一段论证` → `下一张图`。文字和图交替穿插。
- ⚠️ **尽量避免**:把一节的分析写成一大段长文,再把这节相关的好几张图**连续堆在段落后面**(文字成坨 + 图成串)。读者读完一大段已疲,后面一串图失去与论点的锚定。
- **例外**:成对、可对照的图(分部增速对比、三情景、毛利率 vs 净利率)允许一行两图并列(见 §八),不必拆成两段中间塞文字。
- **判断标准**:一张图出现前,问"它前面这段文字有没有讲清楚它说明什么"。有 → 混排到位;没有(图是凭空冒出来的) → 补一段引出文字,或把图挪到对应论点旁。

这是**软建议**:鼓励混排、给读者图文交替的节奏,不是每段都强制配一图。论证密集处可以连写两段再上图,但不要让"一坨文字 + 一串图"成为常态。

## 五、图型对照(沿用 ICA 阶段 4,加 ECharts type)

| 图型 | 适用 | ECharts type |
|---|---|---|
| 折线 | 趋势/时间序列(营收、利润率、股价) | `line` |
| 堆叠面积 | 收入按产品拆分 | `line`+`stack`+`areaStyle` |
| 堆叠柱状 | 收入按地区/业务线占比/季度 | `bar`+`stack` |
| 分组柱状 | 同业横向对比 | `bar` |
| 饼图/环形 | 占比/份额 | `pie`(环形 `radius:['40%','70%']`) |
| 散点/气泡 | 增长 vs 估值、竞争定位、物理锚传导 | `scatter` |
| 热力图 | DCF 敏感性、相关性矩阵 | `heatmap`+`visualMap` |
| 横条 | 估值足球场、同业排名、市场份额 | `bar` horizontal+`markLine` |
| 瀑布 | 收入桥、利润率分析、DCF 拆解 | `bar`(waterfall 拼接) |
| 有向流程图 | 线性因果传导链(物理锚→收入→利润、传导机制) | `graph`(layout:none + 箭头) |
| 桑基 | 多对多流量分配/分流合流(多分部汇入总营收、利润在多产品间分配) | `sankey` |
| 时间轴/散点 | 技术代际、产品路线图、关键事件 markPoint | `line`/`scatter`+`markPoint` |

## 六、推荐图谱(40-70 张,按四章,沿用 ICA 阶段 4)

图清单和取数映射以 ICA [`references/阶段4-图表生成.md`](../../initial-coverage-advanced/references/阶段4-图表生成.md) 为准。按报告四章分配:

| 章 | 推荐图(服务论证,够 40-70 即可) |
|---|---|
| 封面/摘要 | 股价多年走势 vs benchmark(双 Y 轴绝对价格、周频,叠阶段标注 markPoint)—— ICA 必备图 |
| 一、公司概况 | 沿革时间轴、股权结构、分部收入演变(堆叠柱)、历史财务趋势(营收/毛利率/净利) |
| 二、行业空间 | 市场规模增长(柱+线)、产业链位置、竞争格局份额(饼/条) |
| 三、核心驱动·壁垒 | 关键业务市场份额、代际/技术路线时间轴、竞争对手定位矩阵(散点)、对标表 |
| 四、盈利预测与估值 | 分部预测(堆叠柱)、三情景关键指标对比(分组柱)、**隐含价阶梯叠历史股价带**、**三情景 EPS×目标倍数→隐含价瀑布**、DCF 敏感性(热力图)、估值足球场(横条)、SOTP、**个股自身历史 forward P/E·P/B band**、vs 卖方共识对比 |

第三章壁垒图 + 第四章估值语义图是论证骨干,优先保证数量与质量。

## 七、模型语义图(估值通电链可视化,第四章必备)

把估值模型的核心传导逻辑画成图(数字取自 valuation-model,见 [估值章接法.md](./估值章接法.md)):

- **隐含价阶梯**:物理锚→营收→利润→多镜头→隐含价逐级传导,每级一台阶标关键假设。叠历史股价带,让读者看见"哪一年长进现价"。
- **物理锚传导图**:业务最底层物理驱动量(产能/出货量/单价)如何向上传导到收入与利润。**线性传导链用 `graph` 有向图(方框+箭头),不要用 sankey**——线性等量链 sankey 会糊成绸缎;只有真正多对多分流合流才用 sankey。`graph` 使用 `layout:"none"`,为每个节点显式给 `x/y`、`symbol:"roundRect"`、`symbolSize`,边用 `edgeSymbol:["none","arrow"]`;仍复用 §一的完整尺寸、标题、tooltip、legend 与 grid 规范。
- **三情景 EPS×目标倍数→隐含价瀑布**:Bear/Base/Bull 三案从 EPS 到隐含价的拆解。

## 八、双图并列(沿用 ICA,Obsidian 无边框二列表格)

成对/可对照的图(分部增速对比、可比公司倍数、三情景、毛利率 vs 净利率、Bear/Base/Bull 隐含价)鼓励**一行两图各占 50% 居中并列**,提升信息密度。Obsidian 用无边框二列表格实现:

```
| <图 A 的 echarts 代码块> | <图 B 的 echarts 代码块> |
|---|---|
```

每图各自带图表 N 标题 + 一行资料来源。单图占整页仍禁止。

## 九、全 inline 40-70 图工程(性能/维护提示)

全 inline ECharts 40-70 张在一个 md 里,体积 50-200KB。工程要点:

- **layout 完整内联**:每张图都显式写 `width:900`、`height:506`、title/tooltip/legend/grid/坐标轴/series;复制 §一骨架再替换业务内容,不调用外部格式化器。
- **改数同步**:估值/财务图的数字只能从 `extract_model_tables.py` 抽取后填入,**不手敲**(手敲必与模型漂移)。业务/行业图数字标来源。
- **Obsidian 渲染**:obsidian-echarts 插件渲染几十个图,首次打开略慢属正常;若实测严重卡顿,可把次要图(份额排名/分部演变)降级为静态,核心交互图(股价/估值足球场/三情景/物理锚传导)保 inline——但默认全 inline。
- **双端**:同一份 md 在 Obsidian(固定像素 width/height)和 xuang.xyz(CSS aspectRatio)都原生渲染。
