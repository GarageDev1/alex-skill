# 手机转发研究长图

## 目标与输出

将长文档转为适合手机阅读和转发的 1080px 宽竖向长图。生成前先根据内容选择单版或双版:

- `single`：单版。用于行业图谱、供应链梳理、扩产信息、知识整理、技术路线、市场结构，以及不含投资建议的中性公司研究。只生成 `input.png`，成图不显示内部/外部标签。
- `dual`：双版。仅用于针对特定可交易标的且包含投资建议、评级、估值、目标价、仓位、交易动作、证券价格判断或内部跟踪阈值的研究。生成 `input-internal.png` 与 `input-external.png`；内部图显示“内部研究版”，外部图不显示受众标签。

是否双版取决于内容。中性的公司业务介绍仍使用 `single`；只有对特定标的形成投资判断时才使用 `dual`。如果材料同时包含事实整理和投资判断，使用 `dual`。如果无法明确判断，询问用户希望单版还是双版，不得自行猜测。

在 `dual` 模式下，不得只根据“是否公开”判断外发安全性。External 默认排除投资建议和证券估值相关信息，即使原始材料来自公开披露。

默认使用简体中文。

## 输出策略判定

选择 `single`：

- 行业、产业链、供应链、扩产、技术、政策、竞争格局或知识图谱；
- 公司业务、产品、技术和经营事实整理，但没有买卖建议、估值或证券价格判断；
- 研究目的只是建立认知、解释事实或呈现公开信息。

选择 `dual`：

- 针对股票、基金、债券、加密资产或其他可交易标的给出买入/持有/卖出、增减仓、推荐或交易动作；
- 包含估值、市值、目标价、目标市值、股价判断、回报空间或安全边际；
- 包含为投资决策服务的 Bull/Base/Bear 情景、内部跟踪阈值、可证伪条件或仓位纪律；
- 用户明确要求内部版和外部版。

仍不清楚时：先问用户“这张图是中性信息整理，还是包含针对具体标的的投资建议，需要内部/外部双版？”得到回答后再创建 JSON。

## Dual 模式的外部版合规规则

生成 JSON 时为每个章节、观点卡和视觉块显式写入 `audience`：

- `both`：允许出现在 Internal 和 External。
- `internal`：仅允许出现在 Internal。

External 采用“白名单优先”：未标为 `both` 的内容一律不进入外部版。渲染前再过一道关键词过滤；命中下列内容的标题、标签、正文或视觉块一律剔除：

- 评级、买入/持有/卖出、增持/减持、推荐、推票、建仓或交易动作；
- 估值、市值、目标价、目标市值、股价、IPO/发行价、PE、P/E、PS、P/S、DCF、IRR、收益率、上行/下行空间、安全边际；
- Bull/Base/Bear 情景、概率加权、隐含价/隐含市值；
- 公司层面的收入、利润、EPS、出货量或财务预测，以及预测期数字（如 `2027E`）。

External 可保留的典型内容：产品线、商业模式、技术路线、供应链、竞争位置、公开披露的历史经营事实、行业结构、政策背景和不涉及证券价格的经营风险。

外部版仍需人工复核：过滤器是防线，不替代法务、合规或持牌机构的审阅。

## 工作流程

1. 读取源文档并建立论证骨架。
   - 提取公司业务、产品线、技术路径、经营数据、行业位置、竞争格局、风险和验证点。
   - 先区分“业务研究”与“投资判断”；不要先写长图再试图删敏感段落。
   - 历史数据必须保留来源日期和口径；金融、市场和时事材料按需要核验时效性。

2. 选择并记录输出策略。
   - 明确写入 `output_policy: "single"` 或 `output_policy: "dual"`。
   - `single`：只写一层完整内容，不需要为了形式添加 `audience`；渲染器会拒绝包含评级、估值、交易建议等投资敏感词的单版输入。
   - `dual`：`both` 只写外部也可公开展示的业务研究事实；`internal` 补充投资判断、盈利预测、估值、评级、价格区间和可证伪阈值。
   - External 不是 Internal 的“删减摘要”；应围绕“公司如何经营、产品为何有竞争力、行业机会与经营风险”自成完整叙事。
   - 标题与副标题使用面向读者的研究标题，不手工写入“内部研究版”“外部公开版”“Internal”“External”等流程文案；`dual` 的内部标签由渲染器统一生成。

3. 创建 JSON。
   - 读取 `references/schema.md`。
   - 每章使用 2-5 张观点卡，每张卡只表达一个判断：`加粗标签 + 一句话 + 关键事实`。
   - 需要承载原始 reference 的口径对照、业务拆分、竞品比较或情景假设时，使用 `blocks` 中的紧凑表格；每表 2-4 列，超出时拆为两张表或两段，不得用小字号直接填入。
   - `dual` 模式下，表格与观点卡必须逐项标注 `audience`；External 会检查表题、表头和每一个单元格，未标 `both` 的表格不外发。
   - `dual` 模式下，任何不确定是否适合外发的内容都标为 `internal`，且不得在 `external_title`、`external_subtitle`、`share.external_*` 中放入敏感内容。

4. 生成研报页面缩略图。
   - 缩略图的作用是吸引阅读的用户加群，缩略图的内容本身并不属于“至关重要”的东西，但是必须有一个可以吸引用户的锚点。
   - 除非用户显式要求省略缩略图，否则必须生成。源材料为 PDF 时，直接用 pdf2image 转图；为 docx 时先通过 win32com 导出 PDF 再用 pdf2image 转图，或使用 LibreOffice headless。如果源材料为其他格式，你先自行思考生成什么样的缩略图是合适的，是可以吸引用户的，然后把你的推荐选项告诉用户，并且征求用户的意见。
   - 将生成的 `page_01.png`、`page_02.png` 等放入输入 JSON 同目录下的 `_page_thumbs/` 子目录。
   - 在 JSON 中加入 `"docx_preview": {"dir": "_page_thumbs", "pages": N}`，`N` 为实际生成的缩略图数量；渲染器每行最多显示 4 张，超出自动换行。
   - 详见 `references/schema.md` 的"研报预览横条"章节。

5. 按 `output_policy` 渲染。
   - 运行：`python scripts/render_mobile_share.py input.json --output-dir outputs`
   - 默认 `--mode auto`：`single` 输出 `input.png`；`dual` 输出 `input-internal.png` 与 `input-external.png`，并保留对应 HTML 以便审阅。
   - JSON 缺少 `output_policy` 时渲染器会停止；应返回判定步骤，必要时询问用户。
   - `--mode single|both|internal|external` 只用于明确覆盖或调试，不要用来绕过输出策略判断。
   - 若仅需 HTML 调试，使用 `--html-only`；正式交付不得使用该参数。

6. 验收成图。
   - 确认每张 PNG 宽度均为 1080px，内容自然向下延伸，不强行压成 9:16。
   - 确认顶部至少 132px 安全区、左右 72px 边距，以及 70px 白色间隔与固定 400px 高的底部品牌栏完整。
   - `single`：确认只生成一张图，且不显示任何内部/外部标签。
   - `dual`：确认内部图显示“内部研究版”，External不显示受众或流程标签；对External再次搜索禁用词，逐项确认不存在评级、推票、估值、价格、收益空间或预测性财务判断。
   - 检查每张表的全部列都在 608px 正文宽度内；表头与单元格在手机等比例缩放后仍可读，且没有横向溢出或被裁切。
   - 顶部右侧与底部右侧必须使用同一二维码；默认读取 `assets/brand.json` 的 `qr_image`，真实二维码需完整可见并实测可扫码。

## 内容与版式标准

- 复用 `reference` 的结构骨架：单条深色竖线贯穿内容，一级章节标签以浅色块直接压在竖线上，观点条与表格由短横线接入竖线。不得将章节标签另开为左侧栏，也不得将正文改成 `01/02` 文章式编号版。
- 顶部标题属于页面主标题，必须从完整内容宽度起排；不得随着竖线正文列缩进。
- 左侧标签只放精炼章节名；完整逻辑由右侧观点条表达。优先使用 6-12 个汉字的 `section.tag`，过长标题必须压缩后再放入标签。
- 每张观点条都必须通过横线连接至竖线；保持扁平、紧凑、无厚重阴影的参考图风格。
- 相邻一级章节不得使用同一 `tone`。即使输入 JSON 连续指定同色，渲染器也会自动切换为下一种协调色；Internal 与 External 分别按实际可见章节重新计算。
- 导出宽度固定为 1080px，高度由内容决定。9:16 仅用于首屏预览，不是内容边界。
- 顶部保留 132px 安全区；左右各 72px；正文与底栏之间保留 70px 白色间隔，底部使用 1080×400px 科技蓝品牌栏，总占用严格为 470px，不得额外追加纯色安全区。
- 1080px 导出下，左侧标签约 26px、观点条正文约 27px、顶部标题约 52px；正文不得小于 22px。
- 每条观点尽量控制在 2-4 行。超过时拆分、合并或缩短，不要靠压小字号容纳。
- 表格是线图的一部分：置于观点条下方，仍通过横线接入竖向主线；优先用于原 reference 中本就存在的关键对照表，避免把所有正文机械表格化。
- 表格最多 4 列；1080px 导出下表内文字不小于 19px，优先使用 2-3 列以保证手机阅读。
- 默认底部使用“完整研报加入智富界交流群”、平台介绍和“扫码获取完整研报”二维码；顶部二维码栏略小于底部二维码栏，二者均使用同一二维码。
- 底栏以独立科技蓝背景、活字和二维码分层渲染，不得把文字或二维码固化进背景图片。品牌资产和文案默认从 `assets/brand.json` 读取，优先于输入 JSON：左上角品牌短句、底部背景、机构介绍、可选 `footer_points` 兼容文案及上下共用二维码均在该文件维护。直接覆盖 `assets/zhifujie-qr.png` 即可替换两处二维码。

## 环境与依赖

- 优先使用 `uv` 管理 Python 依赖（`uv pip install playwright pdf2image Pillow`）；仅当环境中没有 `uv` 时回退到 `pip`。首次使用还需 `playwright install chromium`。
- 在Windows环境下应该遵循Shell规范，你需要按以下优先级使用Shell：Git Bash >> PowerShell 7 >> PowerShell 5 >> Cmd，永远优先使用前面的Shell，除非当前环境没有该Shell。
- Windows环境UTF-8编码规范：运行任何 Python 脚本前，先设置环境变量确保 UTF-8：PowerShell 中 `$env:PYTHONUTF8="1"`，Bash 中 `export PYTHONUTF8=1`。不设此变量，中文内容会因系统默认编码（GBK/cp936）变为乱码。永远提前思考，这个操作是否可能因为编码问题出错或者乱码？
- 禁止使用 `type`、`Get-Content` 或无编码参数的 shell 命令读取含中文的源文件；应使用 Read 工具或 Python `open(path, encoding="utf-8")` 读取。
